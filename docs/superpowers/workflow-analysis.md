# Superpowers-CN 技能工作流全景分析

> 基于对 skills/ 目录下全部 9 个技能的完整源码分析，梳理端到端开发流程、技能调用关系和子代理拓扑。

---

## 一、技能清单总览

| # | 技能名称 | 类型 | 触发时机 | 是否拉起子代理 |
|---|---------|------|---------|--------------|
| 1 | using-superpowers | 元技能 | 每次对话开始、任何响应之前 | 否（调度器角色） |
| 2 | brainstorming | 流程技能 | 任何创造性工作之前（功能、组件、修改行为） | 是（规格文档审查员） |
| 3 | writing-plans | 流程技能 | 有规格说明/需求，准备编写实现计划 | 是（计划文档审查员） |
| 4 | subagent-driven-development | 实现技能 | 有实现计划，任务基本独立 | 是（实现者 + 规格审查者 + 代码质量审查者 + 最终审查者） |
| 5 | requesting-code-review | 质量技能 | 任务完成/功能实现后、合并前 | 是（code-reviewer 子代理） |
| 6 | receiving-code-review | 响应技能 | 收到代码审查反馈后 | 否（自身处理反馈） |
| 7 | test-driven-development | 纪律技能 | 所有代码变更（新功能/修bug/重构） | 否（行为规范） |
| 8 | systematic-debugging | 流程技能 | 遇到 bug/测试失败/异常行为 | 否（调试框架） |
| 9 | verification-before-completion | 纪律技能 | 宣称任何完成/成功之前 | 否（验证规范） |

## 三、各阶段详细分析

### 阶段 1：头脑风暴（brainstorming）

**使用技能：** `brainstorming`

**拉起的子代理：** 无（自检是内联完成，非子代理）

**流程：**

```
用户请求
    │
    ▼
┌─────────────────────┐
│ 1. 探索项目上下文      │  ← 关键词定向搜索代码仓
│    （Grep/Glob/Read） │     识别现有模式、技术栈、影响范围
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ 2. 提出澄清问题       │  ← 只问代码无法回答的问题
│    （每次一个）        │     意图/业务规则/优先级/外部约束
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ 3. 提出 2-3 种方案    │  ← 附权衡分析和推荐
│    + 展示设计          │     按复杂度分节，逐节获批准
└──────────┬──────────┘
           ▼
    用户批准设计？
     │否          │是
     └─(修改)─────┘
                   ▼
┌─────────────────────┐
│ 4. 编写 SPEC.md      │  → docs/superpowers/specs/YYYY-MM-DD-<topic>-spec.md
│    （WHAT：需求+场景） │     格式：ADDED/MODIFIED Requirements + WHEN/THEN
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ 5. SPEC.md 自检       │  ← 场景完整性、可测试性、一致性、范围、模糊性
│    （内联修复）        │     发现问题直接修，无需重审
└──────────┬──────────┘
           ▼
    用户审查 SPEC.md？
     │修改          │批准
     └─(修改)───────┘
                    ▼
┌─────────────────────┐
│ 6. 编写 DESIGN.md    │  → docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md
│    （HOW：技术设计）   │     Context / Goals / Decisions / Architecture / Risks
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ 7. DESIGN.md 自检     │  ← 需求追溯、占位符、一致性、孤立设计、可构建性
│    （内联修复）        │
└──────────┬──────────┘
           ▼
    用户审查 DESIGN.md？
     │修改          │批准
     └─(修改)───────┘
                    ▼
         过渡到 writing-plans
```

**关键文件模板：** `brainstorming/spec-document-reviewer-prompt.md`（注意：此模板存在但当前 brainstorming 流程中自检是内联完成的，该模板用于可选的独立审查场景）

**产出物：**
- `SPEC.md` — 需求规格（WHAT）+ WHEN/THEN 场景
- `DESIGN.md` — 技术设计（HOW）+ 架构决策

### 阶段 2：编写计划（writing-plans）

**使用技能：** `writing-plans`

**拉起的子代理：** 无（自检是内联完成）

**流程：**

```
SPEC.md + DESIGN.md
         │
         ▼
┌─────────────────────────┐
│ 1. 读取输入文档            │  ← 优先读 SPEC.md，再读 DESIGN.md
│    按优先顺序              │     向后兼容：仅有 design.md 时按旧格式处理
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│ 2. 定义文件结构            │  ← 列出将创建/修改的文件及职责
│    锁定分解决策            │     设计边界清晰的单元，每文件一职责
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│ 3. 拆分为小步骤任务        │  ← 每步 2-5 分钟
│    （TDD 模式可选）         │     TDD: 测试→验证失败→实现→验证通过→提交
│                           │     非TDD: 实现→编译→提交
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│ 4. 自检计划               │  ← 规格覆盖度 / 占位符扫描 / 类型一致性
│    （内联修复）            │     测试步骤完整性
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│ 5. 保存计划               │  → docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md
│    + 执行交接              │     头部标注：使用 subagent-driven-development 执行
└─────────────────────────┘
```

**计划文档格式要求：**
- 头部包含：目标、架构、技术栈、规格/设计路径
- 每个任务精确到文件路径和代码
- 禁止占位符（"待定"、"TODO"、"类似任务 N"）
- 步骤必须包含完整代码块

**产出物：**
- `PLAN.md` — 实现计划，头部标注 `subagent-driven-development`

**关键文件模板：** `writing-plans/plan-document-reviewer-prompt.md`（同上，可选的独立审查模板）

### 阶段 3：开发 — 子代理驱动 + 两阶段审查（subagent-driven-development）

**使用技能：** `subagent-driven-development`（核心实现引擎）

**拉起的子代理（每个任务最多 3 个，所有任务完成后 1 个）：**

| 子代理 | 提示模板 | 角色 | 模型建议 |
|--------|---------|------|---------|
| **实现子智能体** | `implementer-prompt.md` | 按步骤实现、测试、提交、自审 | 机械任务→便宜模型；集成→标准模型 |
| **规格合规审查子智能体** | `spec-reviewer-prompt.md` | 验证实现是否匹配规格（不多不少） | 最强模型 |
| **代码质量审查子智能体** | `code-quality-reviewer-prompt.md` | 验证编译通过、测试通过 | 标准模型 |
| **最终代码审查子智能体** | （无独立模板，使用 requesting-code-review/code-reviewer.md） | 审查整体实现质量 | 最强模型 |

**完整流程：**

```
读取计划文件，提取全部任务的完整文本和上下文
创建 TodoWrite（所有任务列表）
            │
            ▼
    ┌─── 还有剩余任务？ ─────────── 否 ──────┐
    │                                         │
   是                                         ▼
    │                              ┌─────────────────────┐
    ▼                              │ 分派最终代码审查子智能体 │
┌───────────────────────────────┐  │ （审查整体实现）        │
│ 分派「实现子智能体」              │  └──────────┬──────────┘
│ 模板：implementer-prompt.md    │             │
│ 上下文：完整任务文本+场景铺设     │             ▼
└───────────────┬───────────────┘     报告完成，验证测试通过
                │                     等待用户指示
                ▼
        实现者有疑问？
         │              │
        是              否
         │              │
         ▼              ▼
  回答问题            ┌─────────────────────────┐
  提供上下文          │ 实现者执行：               │
  重新分派            │  1. 严格按步骤实现         │
                     │  2. 编写测试               │
                     │  3. 验证功能               │
                     │  4. 提交 git commit        │
                     │  5. 自审（完整性/质量/纪律）│
                     │  6. 汇报状态               │
                     └────────────┬──────────────┘
                                  │
                    ┌─────────────┤
                    │ 状态：
                    │ DONE → 继续
                    │ DONE_WITH_CONCERNS → 读疑虑后继续
                    │ NEEDS_CONTEXT → 补充上下文重新分派
                    │ BLOCKED → 评估原因，换模型/拆任务/上报
                    └─────────────┘
                                  │
                         (DONE/DONE_WITH_CONCERNS)
                                  │
                                  ▼
                 ╔═════════════════════════════════╗
                 ║   第一阶段审查：规格合规性         ║
                 ║   spec-reviewer-prompt.md       ║
                 ╠═════════════════════════════════╣
                 ║                                 ║
                 ║   分派规格审查子智能体              ║
                 ║   ↓                             ║
                 ║   审查者独立验证（不信任报告）：   ║
                 ║   • 缺失的需求？                 ║
                 ║   • 多余/不需要的工作？           ║
                 ║   • 理解偏差？                   ║
                 ║   • SPEC.md 每个 WHEN/THEN 覆盖？║
                 ║   ↓                             ║
                 ║   结果：                        ║
                 ║   ✅ 符合规格 → 进入第二阶段      ║
                 ║   ❌ 发现问题 → 实现者修复        ║
                 ║              ↓                  ║
                 ║         重新审查（循环）          ║
                 ╚═════════════════════════════════╝
                                  │
                         (规格审查通过)
                                  │
                                  ▼
                 ╔═════════════════════════════════╗
                 ║   第二阶段审查：代码质量           ║
                 ║   code-quality-reviewer-prompt.md║
                 ╠═════════════════════════════════╣
                 ║                                 ║
                 ║   ⚠️ 仅在规格合规性审查通过后分派   ║
                 ║                                 ║
                 ║   分派代码质量审查子智能体          ║
                 ║   ↓                             ║
                 ║   执行两项检查：                  ║
                 ║   1. 编译检查：运行构建命令        ║
                 ║   2. 测试检查：运行单元测试        ║
                 ║   ↓                             ║
                 ║   结果：                        ║
                 ║   ✅ 编译+测试均通过 → 任务完成    ║
                 ║   ❌ 任一失败 → 实现者修复         ║
                 ║              ↓                  ║
                 ║         重新审查（循环）          ║
                 ╚═════════════════════════════════╝
                                  │
                         (代码质量通过)
                                  │
                                  ▼
                    在 TodoWrite 中标记任务完成
                                  │
                                  ▼
                        ◀── 回到「还有剩余任务？」
```

**子代理状态处理：**

| 状态 | 含义 | 处理方式 |
|------|------|---------|
| `DONE` | 实现完成，无疑虑 | 直接进入规格审查 |
| `DONE_WITH_CONCERNS` | 完成但有疑虑 | 阅读疑虑后决定是否继续 |
| `NEEDS_CONTEXT` | 缺少信息 | 补充上下文后重新分派 |
| `BLOCKED` | 无法继续 | 换更强模型/拆分任务/上报人类 |

**红线规则：**
- 规格合规审查**必须先于**代码质量审查（顺序不可颠倒）
- 审查者发现问题 → 实现者修复 → 审查者重新审查（不可跳过循环）
- **不并行分派**多个实现子智能体（会冲突）
- 子智能体**不读取计划文件**（控制者提供完整文本）

### 阶段 4：代码审查（requesting-code-review + receiving-code-review）

这两个技能是**互补关系**，用于不同场景：

#### requesting-code-review（请求代码审查）

**触发时机：** 你（或子代理驱动开发流程）**主动发起**代码审查时

**典型场景：**
- `subagent-driven-development` 的最终代码审查阶段
- 完成重要功能后，合并到 main 之前
- 卡住时，需要换个视角
- 重构之前，建立基线

**拉起的子代理：**

| 子代理 | 提示模板 | 角色 |
|--------|---------|------|
| **code-reviewer 子代理** | `requesting-code-review/code-reviewer.md` | 审查代码的生产就绪程度 |

**流程：**

```
需要审查代码
     │
     ▼
┌──────────────────────┐
│ 1. 获取 git SHA       │  BASE_SHA（起始）+ HEAD_SHA（当前）
│    git rev-parse      │
└──────────┬───────────┘
           ▼
┌──────────────────────┐
│ 2. 派遣 code-reviewer │  Task tool (general-purpose)
│    子代理              │  填写 code-reviewer.md 模板：
│                       │  • WHAT_WAS_IMPLEMENTED
│                       │  • PLAN_OR_REQUIREMENTS
│                       │  • BASE_SHA / HEAD_SHA
│                       │  • DESCRIPTION
└──────────┬───────────┘
           ▼
┌──────────────────────┐
│ 3. 审查维度            │
│    • 代码质量（DRY等）  │
│    • 架构合理性         │
│    • 测试覆盖          │
│    • 需求满足度         │
│    • 生产就绪度         │
└──────────┬───────────┘
           ▼
┌──────────────────────┐
│ 4. 分级反馈            │
│    Critical → 立即修复 │
│    Important → 先修复  │
│    Minor → 记录后处理  │
└──────────────────────┘
```

#### receiving-code-review（接收代码审查）

**触发时机：** 你**收到**来自他人的代码审查反馈时

**典型场景：**
- PR 收到审查评论
- 搭档给出代码反馈
- 外部审查者提供改进建议

**拉起的子代理：** 无（自身处理反馈）

**流程：**

```
收到代码审查反馈
        │
        ▼
┌──────────────────────────┐
│ 1. 阅读                   │  完整阅读，不急于反应
└────────────┬─────────────┘
             ▼
┌──────────────────────────┐
│ 2. 理解                   │  用自己的话复述需求（或提问）
└────────────┬─────────────┘
             ▼
┌──────────────────────────┐
│ 3. 验证                   │  对照代码库实际情况检查
│    • 技术上正确吗？        │  会破坏现有功能吗？
│    • 审查者了解完整上下文？ │  是否有遗留/兼容性原因？
└────────────┬─────────────┘
             ▼
┌──────────────────────────┐
│ 4. 评估                   │  对这个代码库来说合理吗？
└────────────┬─────────────┘
             ▼
┌──────────────────────────┐
│ 5. 回应                   │  技术性确认 OR 有理有据的反驳
│    ❌ 禁止敷衍附和         │  "你说得太对了！" → 禁止
│    ✅ 直接行动             │  "已修复。[说明]"
└────────────┬─────────────┘
             ▼
┌──────────────────────────┐
│ 6. 实施                   │  一次一项，逐个测试
│    顺序：                  │
│    ① 阻塞性（崩溃/安全）   │
│    ② 简单修复              │
│    ③ 复杂修复              │
└──────────────────────────┘
```

**两者的关系总结：**

```
requesting-code-review          receiving-code-review
       │                               │
       │ 你主动请别人审查               │ 别人给你审查反馈
       │                               │
       ▼                               ▼
  派遣 code-reviewer              自身处理反馈
  子代理执行审查                   （无子代理）
       │                               │
       ▼                               ▼
  得到审查结果                     验证→评估→实施
  （Critical/Important/Minor）     或反驳
```

### 阶段 5：完成（verification-before-completion）

**使用技能：** `verification-before-completion`

**拉起的子代理：** 无

**此技能贯穿整个流程，不仅限于最后阶段。** 在以下每个节点都必须使用：

| 触发节点 | 验证内容 |
|---------|---------|
| 头脑风暴完成 SPEC.md | 场景完整性、一致性 |
| 编写计划完成 | 规格覆盖度、类型一致性 |
| 每个子代理任务完成 | 测试通过、编译通过 |
| 代码审查反馈修复后 | 修复正确、无回归 |
| 最终完成 | 全部测试通过、构建成功 |

**验证门控函数：**

```
宣称任何完成之前：
  1. 确定：什么命令能证明这个结论？
  2. 运行：执行完整命令（重新运行）
  3. 阅读：完整输出 + 退出码 + 失败数
  4. 验证：输出是否支持结论？
  5. 只有这时：才能做出结论
```

**子代理调用统计（假设计划有 N 个任务）：**

| 子代理类型 | 每任务调用次数 | 总计调用次数 | 备注 |
|-----------|-------------|------------|------|
| 实现子智能体 | 1（+修复循环） | N + 修复次数 | 可通过 NEEDS_CONTEXT/BLOCKED 增加重试 |
| 规格合规审查子智能体 | 1（+修复后重审） | N + 重审次数 | 固定在代码质量审查之前 |
| 代码质量审查子智能体 | 1（+修复后重审） | N + 重审次数 | 固定在规格审查通过之后 |
| 最终代码审查子智能体 | 1 | 1 | 所有任务完成后 |
| code-reviewer 子代理 | 按需 | 0~多次 | requesting-code-review 触发 |

**典型 5 任务项目最小子代理调用：** 5×3 + 1 = **16 次**（不含修复循环）

## 五、纪律技能的贯穿方式

`test-driven-development`、`systematic-debugging`、`verification-before-completion` 三个纪律技能不以独立阶段出现，而是嵌入到每个阶段中：

```
┌─────────────────────────────────────────────────────────────┐
│                      TDD 纪律                                │
│                                                              │
│  头脑风暴阶段：无直接应用                                       │
│  编写计划阶段：每个任务默认包含 TDD 步骤（可选关闭）              │
│  开发阶段：  实现子代理严格按红→绿→重构执行                      │
│  代码审查：  审查者检查测试是否真正验证行为                       │
│  完成阶段：  验证测试命令实际通过                                │
├─────────────────────────────────────────────────────────────┤
│                    系统调试纪律                                │
│                                                              │
│  触发条件：任何 bug、测试失败、异常行为                          │
│  流程：根因调查 → 模式分析 → 假设验证 → 实现修复                 │
│  可在任何阶段触发（开发/审查/验证）                               │
├─────────────────────────────────────────────────────────────┤
│                   完成前验证纪律                                │
│                                                              │
│  在任何 "完成" 声明之前必须运行验证命令                          │
│  包括：子代理任务完成、审查通过、bug 修复、最终交付               │
│  铁律：没有新鲜验证证据，不许宣称完成                             │
└─────────────────────────────────────────────────────────────┘
```

## 六、requesting-code-review 与 receiving-code-review 的使用区别

| 维度 | requesting-code-review | receiving-code-review |
|------|----------------------|---------------------|
| **角色** | 审查的**发起方** | 审查反馈的**接收方** |
| **触发** | 你需要别人审查你的代码 | 别人给你审查意见 |
| **子代理** | 派遣 code-reviewer 子代理 | 无子代理，自身处理 |
| **输入** | git diff 范围 + 需求/计划 | 审查反馈（Critical/Important/Minor） |
| **输出** | 分级问题列表 + 合并评估 | 验证后的实施决策（接受/反驳/澄清） |
| **典型场景** | 功能完成后主动请求审查 | PR 收到评论后处理反馈 |
| **与子代理开发的关系** | 最终审查阶段的底层机制 | 不直接参与子代理开发流程 |
| **核心原则** | 早审查，勤审查 | 先验证再实施，技术严谨优先 |

**在 subagent-driven-development 中的定位：**
- `requesting-code-review` 的 `code-reviewer.md` 模板是最终代码审查子代理的基础
- `receiving-code-review` 不直接参与子代理开发流程，但当用户（搭档）给出反馈时使用

## 七、完整时序示例

以一个 3 任务项目为例，展示完整时序：

```
时间 ──────────────────────────────────────────────────────────▶

T1  用户："我想给系统添加导出功能"
    │
    ├─▶ [using-superpowers] 检查适用技能
    │   └─▶ [brainstorming] 进入头脑风暴
    │       ├─ 探索代码仓 → 提问 → 方案选择 → 设计展示
    │       ├─ 用户批准设计
    │       ├─ 编写 SPEC.md → 自检 → 用户审查 → 通过
    │       └─ 编写 DESIGN.md → 自检 → 用户审查 → 通过
    │
T2  [writing-plans] 编写计划
    │   ├─ 读取 SPEC.md + DESIGN.md
    │   ├─ 定义文件结构
    │   ├─ 拆分为 3 个任务（TDD 步骤）
    │   ├─ 自检计划
    │   └─ 保存 PLAN.md
    │
T3  [subagent-driven-development] 开始执行
    │
    ├─── 任务 1 ───
    │   ├─▶ [实现子代理] 实现任务 1 → DONE
    │   ├─▶ [规格审查子代理] ✅ 符合规格
    │   └─▶ [代码质量子代理] ✅ 编译+测试通过
    │
    ├─── 任务 2 ───
    │   ├─▶ [实现子代理] 实现任务 2 → DONE
    │   ├─▶ [规格审查子代理] ❌ 缺少进度报告
    │   ├─▶ [实现子代理] 修复 → 重新提交
    │   ├─▶ [规格审查子代理] ✅ 现在符合
    │   ├─▶ [代码质量子代理] ❌ 魔法数字
    │   ├─▶ [实现子代理] 提取常量 → 重新提交
    │   └─▶ [代码质量子代理] ✅ 通过
    │
    ├─── 任务 3 ───
    │   ├─▶ [实现子代理] 实现任务 3 → DONE
    │   ├─▶ [规格审查子代理] ✅ 符合规格
    │   └─▶ [代码质量子代理] ✅ 通过
    │
    └─▶ [最终代码审查子代理] ✅ 可以合并
        │
T4      ├─▶ [verification-before-completion] 全量验证
        │   └─▶ 运行全部测试 → 通过 ✅
        │
T5      └─▶ [可选] requesting-code-review
            └─▶ [code-reviewer 子代理] 最终审查
                │
                └─▶ 用户合并 PR
```

## 八、总结

### 核心设计哲学

1. **隔离上下文**：每个子代理获得精心构造的独立上下文，不继承会话历史
2. **两阶段审查**：规格合规性（建了什么）→ 代码质量（建得好不好），顺序不可逆
3. **不信任报告**：审查者独立阅读代码验证，不信任实现者的自我报告
4. **验证驱动**：所有完成声明必须有新鲜的验证证据支撑
5. **质量关卡**：自审 → 规格审查 → 代码质量审查，层层递进

### 技能依赖链

```
using-superpowers（元调度，贯穿始终）
    │
    ▼
brainstorming
    │ → SPEC.md + DESIGN.md
    ▼
writing-plans
    │ → PLAN.md
    ▼
subagent-driven-development
    │ → 实现代码
    ▼
requesting-code-review（可选，合并前审查）
    │
    ▼
verification-before-completion（贯穿始终）

横向贯穿：
    test-driven-development（所有代码变更）
    systematic-debugging（遇到问题时）
    receiving-code-review（收到审查反馈时）
```
